# セッション内容を技術記事化

このコマンドは、**今回の作業セッション**を対外向けの技術記事として整理する。
センシティブ情報は必ず除外し、再現可能な内容にする。

## 1. 情報収集

以下を収集して、記事の材料にする。

```bash
ROOT="$(git rev-parse --show-toplevel)"
DATE="$(date +%Y-%m-%d)"
OUT_DIR="$ROOT/docs/articles"
OUT_FILE="$OUT_DIR/${DATE}_session-article.md"

git status -sb
git log -5 --oneline
git diff --stat
```

必要に応じて以下も確認：
- 参照したドキュメント（`docs/` など）
- 変更したコマンド/設定（`/commands`, `/rules`, `.claude`, `.gemini` 等）
- 実行したテストや検証コマンド

## 2. 記事テンプレート作成

```bash
mkdir -p "$OUT_DIR"
cat <<'EOF' > "$OUT_FILE"
# タイトル（具体的に）

## 背景 / 課題
- なぜ必要だったか
- どんな問題があったか

## 取り組んだこと
- 実施内容の要点
- 重要な判断や方針

## 実装・設定のポイント
- 具体的な設定や構成（必要な範囲で）
- 重複回避や運用上の工夫

## 成果
- 何が改善されたか
- 何が明確になったか

## 学び / 次のアクション
- 得られた知見
- 今後の課題
EOF
```

## 3. 内容の埋め込み

- 収集した情報をもとに、**対外向けに分かる言葉**で本文を埋める。
- 社外秘/個人情報/未公開URLなどは省く。
- 箇条書き中心で簡潔に。必要なら図/リンクを追記。

## 4. 出力
obsidianのINBOXに出力すること。
`$OUT_FILE` をユーザーに提示すること。

## 5. 文体プロンプト
```
あなたは「k4h4shi」風の文体で日本語文章を書く。狙いは“結論を出す”よりも、思考の途中経過を切れ味よく記録すること。

# 文体ルール
- 口語の独白調で書く（「〜と思う」「〜な気がする」「なんか」「〜なんよな」）。
- 断定は避け、確信度を添える（多分/仮説/経験上/〜かもしれない）。必要なら「確証性バイアスかも」と自分にツッコむ。
- 短文＋改行多め。1段落は2〜4行目安。ときどき一行メモも混ぜる。
- 具体例→抽象化→別の具体例、の往復を入れる（レイヤーを上げ下げする）。
- アナロジー（体感の比喩）を1つ以上入れる。
- 技術（AI/開発/プロトコル/テスト等）と、価値観・倫理・文化（全体最適/無関心/民主化等）を自然に接続する。
- たまに厨二っぽい固有名詞・言い回しを“わかってて”差し込む（やりすぎない）。
- Obsidian記法を許容：[[Wikiリンク]]、箇条書き、引用(>)、URL。必要なら見出しも可。

# 構成テンプレ（状況に応じて選ぶ）
A) ショートメモ：

- 何を考えてるか宣言

- 抽象化（構造・フレーム・システムの話）
- 反対側の見方/懸念を一言

# 出力条件

それでは上記条件で、k4h4shiっぽいメモを書いて。
```
