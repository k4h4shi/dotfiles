# レトロ（収集→峻別→改善Issue化）

目的は「次スプリントで同じ詰まりを踏まない」こと。人の注意ではなく **仕組み化** に落とす。

あなたの開発プロセス上の位置づけ:
- デリバリー（スプリント）の終わりに、各 Issue（/fix単位）の学びを集約する
- 出力（成果物）は **改善Issue**（=ディスカバリーの入力）としてGitHubに残し、次スプリントへ繋げる

## 1. レトロの回収
- 各レーン（worktree/Issue）ごとに「何が起きたか」を短く集める
- 可能ならPR/Issue番号、失敗ログ、再現条件を添える
- できれば各 /fix の最後に、子Issueへ **セッションレトロ（短文でOK）** を残す（後で拾いやすくする）

## 2. 出力テンプレ（各レーン）
- Session Summary（終えたこと）
- What Went Well（再現性ある良かった点）
- Pain Points（詰まった点：根本原因の仮説まで）
- Suggested Improvements（改善案：仕組み化）
- Action Items（次にやること）

## 3. 峻別ルール（本当にやる改善は1〜3件）
### 採用
- 複数Issueで再発している
- “人の注意”ではなく“仕組み”で潰せる
- repo内に落とせる（個人設定に閉じない）
- Deliveryと衝突しにくい

### 見送り
- 1回限り/原因不明
- 改善のための改善（運用に乗らない）
- 個人のIDE設定に閉じる

## 4. 改善Issue化（統合して作る）
改善は細かく分けすぎず、1つのIssueに「今回やる範囲」を絞って書く（スプリントゴールのノイズを増やさない）。

最低限含める:
- 背景（レトロのPain Pointsリンク）
- 目的
- 対応範囲（今回やる/やらない）
- 完了条件

補足（重要）:
- 改善Issueは **ディスカバリー側の成果物**（次スプリントの入力）
- 次スプリントで /fix 投入するなら、**セル=Issue**（実行可能な粒度）に落とす
- トラッカーを作る場合は「/fixしない」を明記し、実行用の子Issueに分割する

## 5. 親Issueへの反映（必須）
スプリントの学びが散逸しないよう、レトロの要点は **スプリント親Issue** にコメントで残す。

- 何が効いたか（再現性のあるもの）
- どこが詰まったか（根本原因）
- 次スプリントでやる改善（1〜3件）
